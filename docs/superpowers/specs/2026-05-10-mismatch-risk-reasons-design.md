# Mismatch-Risk Reasons Inline Design

Branch: `claude/fix-website-e2e-Rj73P`

## Goal

Show users WHY each job is flagged as mismatch-risk by rendering the
existing `risks` strings (e.g., `role_mismatch`, `location_mismatch`,
`skills_not_confirmed`) inline next to the job in the dashboard's
`<details class="mismatch-risks">` block. Today the user only sees title
and score and has to click into the job detail page to find the rule
that fired.

## Context

`JobFitResult` (`ml_job_swarm/filtering.py:59`) carries `risks: tuple[str, ...]`
populated by the rules engine — the values are exactly the per-rule flags
the user needs to make sense of the mismatch group. The dashboard's
mismatch-risks list (`ml_job_swarm/web/templates/dashboard.html:144`)
currently renders `{{ job.title }} - {{ job.fit_score }}` only. The fix
is purely presentational: surface what's already in memory.

## V1 Scope

- In `dashboard.html` mismatch-risks `<li>`, render the job's `risks` as
  a comma-separated list inside a `<span class="job-risks">` next to the
  fit score.
- When `risks` is empty, render nothing extra (no placeholder).

## Out Of Scope

- No human-readable mapping of rule codes to friendly labels (e.g.,
  `role_mismatch` → "Role does not match"). That's a follow-up styling
  question.
- No reasons list (positive matches) — only risks, since this section is
  about why the match was downgraded.
- No changes to the hidden-jobs `<details>` block, the job-detail page,
  or the saved jobs page.

## Data And Safety

`risks` is generated by the local rules engine from preferences and job
metadata; no resume content, no LLM output. Output is Jinja-escaped.
